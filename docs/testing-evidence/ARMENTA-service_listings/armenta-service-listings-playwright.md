## Test Scenario Documentation

### Functionality Tested

Talent dashboard service listings in BRaket, specifically:

- Creating service listings
- Viewing created service listings
- Editing existing service listings
- Deleting service listings
- Validating invalid service form input
- Checking access control for unauthenticated and non-talent users

### Objective Of The Test

The objective is to verify that the Service Listings feature works correctly through automated browser testing. The tests confirm that valid service listing actions are reflected in both the user interface and the database, while invalid or unauthorized actions are blocked.

### Steps / Procedure

1. Run the Playwright E2E test suite.
2. Create dedicated test users through the automated test setup.
3. Sign in as a talent user.
4. Open the talent dashboard service listings page.
5. Create a new service listing using valid service data.
6. Verify that the created service appears in the UI and exists in the database.
7. Edit a seeded service listing.
8. Verify that the edited title and price are shown in the UI and persisted in the database.
9. Delete a seeded service listing.
10. Verify that the deleted service disappears from the UI and is removed from the database.
11. Submit invalid service data with a minimum price greater than the maximum price.
12. Verify that validation prevents submission and no database row is created.
13. Visit the service management page while unauthenticated.
14. Verify that the unauthenticated user is redirected to the login page.
15. Sign in as a client user.
16. Verify that the client user cannot access talent service listing management.

### Expected Result

- A talent user can create a valid service listing.
- The created listing appears on the My Services page.
- The created listing is saved in the database.
- A talent user can edit an existing service listing.
- Edited service details are reflected in both the UI and database.
- A talent user can delete an existing service listing.
- Deleted service listings are removed from both the UI and database.
- Invalid price ranges are blocked before submission.
- Invalid service data does not create a database row.
- Unauthenticated users are redirected to login.
- Client users cannot access talent service listing management.

## Reflection / Findings

### Issues Encountered

- Some Playwright locators initially matched more than one element, such as the password field also matching the show-password button.
- The service form initially stayed on the create page because the test user fixture did not satisfy the app's talent onboarding finalization rules.
- The price display assertion originally expected a different format than the actual UI output.

### Warnings / Errors Detected

- Next.js displayed a middleware deprecation warning during test execution.
- React displayed hydration mismatch warnings during login page rendering.
- ESLint reported a pre-existing warning in `app/_home/home-features-section.tsx` for an unused `semantic` import.
- These warnings did not cause the Playwright tests to fail.

### Bugs Discovered

No blocking Service Listings application bug was confirmed after the final test setup was corrected. The issues found were mainly test-fixture and locator problems, which were fixed during implementation.

### Improvements Made After Testing

- The original single CRUD test was split into four focused automated test scripts.
- Database verification was added for create, edit, delete, and invalid-input cases.
- Security and access-control tests were added.
- Screenshots were attached to the Playwright report as external evidence.
- A CI/CD workflow was added through GitHub Actions for automated Playwright pipeline testing.

### Lessons Learned From Automation Testing

- Automated tests should verify both what users see and what the database stores.
- Test users must satisfy the same business rules as real users.
- Strict locators help reveal ambiguous UI labels and improve test reliability.
- Validation tests are important because they confirm that invalid data does not reach the database.
- Access-control tests are important for protecting role-specific dashboard features.
